# TextResurrector MVP  最小可行原型说明书及技术实现方案

## 一、核心原则

· 聚焦单一核心功能，快速验证用户需求

· 核心功能：上传AI图→框选乱码区域→输入文字→渲染文字（低成本视觉融合）→下载修复图

· 舍弃功能：自动识别文字（OCR）、高级光影融合、付费功能、Discord 机器人、专业修图的 OpenCV inpaint

· 原型允许快速迭代，保留错误/冗余记录，确保AI协作可复用

**·** ***\*核心价值不跑偏\****：重点是 “修复的文字能自然融入原图”，而不是单纯让操作步骤少。

## 二、用户操作流程

1. 把 AI 生成的乱码图片（JPG/PNG 格式）传到网页上
2. 输入正确文字（支持多行）
3. 系统自动在框选区域渲染文字，做完简单的融合处理后，直接下载修复后的图片

### 渲染效果低成本优化

· 字体：思源黑体，后续可增加3-5种字体选择

· 字号：动态计算（选框高度 × 0.8），可微调

· 颜色+描边：黑白反色 + 白色描边，保证文字清晰

· 半透明底色：吸取框选周围平均颜色

· 边缘模糊：高斯模糊处理文字边缘，radius = max(1, int(字号/12))

## 三、技术实现方案

### 前端

· HTML5（搭网页结构）+ JavaScript（做交互，比如框选、上传）

· 更轻量的 Canvas 插件，实现鼠标圈选乱码区域的功能，精准定位要修的地方

· 多行输入框（textarea），换行符由后端渲染

· 坐标映射：你在网页上看到的图片是缩放后的，系统会把你框选的网页显示坐标，精准转换成图片原始坐标，保证文字渲染在正确的位置，不会偏。

### 后端

· Python 3.9+（基础语言）+ Flask 框架（搭 API 接口，接收前端的图片、框选坐标、文字信息）

· Pillow渲染文字（动态字号、描边、半透明底色、边缘高斯模糊）

· 可选OpenCV inpaint（非核心）

· 性能：1080p图片在8G内存CPU下<5秒

### 渲染流程

1. 生成蒙版:根据用户框选的坐标，在图片上生成一个 “蒙版区域”（只处理这个区域，不碰其他地方）
2. 接收用户输入文字
3. 用 Pillow 库，在蒙版区域里渲染文字，同时应用自动字号、颜色描边、半透明底色的效果
4. 对渲染好的文字边缘做高斯模糊，让融合度更高
5. 把修复后的图片传回前端，用户直接下载。

### API接口示例

POST /repair
Request: multipart/form-data
\- image: 文件(JPG/PNG)
\- bbox: 框选坐标 x1,y1,x2,y2
\- text: 用户输入文字
Response: 修复图片 (JPEG/PNG)
Errors:
\- 400: 坐标越界或非图片
\- 500: 渲染失败

### 性能限制

· 图片短边 ≤1080px,避免大图片处理太慢；

· 限制并发请求数,避免 Flask 框架（单线程）被阻塞，导致所有人都卡

· 如果处理时间稍长，前端会显示加载动画，告诉用户 “正在处理”

## 四、版本管理

· v0.1：核心流程跑通，文字自然融入，3-5名用户测试

· v0.2：优化网页界面，增加 “保存上次输入文字” 的小功能，提升体验；

· v0.3：加付费问卷、用户操作反馈埋点、数据分析，开始验证用户愿不愿意花钱。

## 五、AI协作流程

· 每次任务前填写任务卡片

· AI生成代码后人工测试通过/失败

· 失败信息反馈AI修正

· 聊天记录存档至docs/chat_logs/

· DECISIONS_LOG.md更新关键决策

## 六、非功能需求

· 隐私：明确告诉用户 “上传的图片是否存储、怎么存储”，消除用户顾虑；

· 错误处理：上传图片失败、渲染文字失败时，给用户明确的文字提示

· 用户体验：实时预览、操作提示、下载格式选择

· 硬件适配：8G内存CPU可运行

## 七、开发前关键动作

1. 技术评审会：确认DoD、坐标映射、多行文本、图片格式
2. 项目脚手架生成：AI辅助生成Flask+前端HTML模板
3. 建立Beta测试反馈圈：找 3-5 个真实的 AI 绘图用户，提前说好测试要求，开发完让他们先测，收集操作中的痛点，为 v0.2 迭代提供数据。

## 八、三个致命缺陷及调整

| 致命缺陷     | 现状                   | 立刻调整                                             |
| ------------ | ---------------------- | ---------------------------------------------------- |
| 需求验证缺失 | 假设用户需要工具       | Landing Page+邮箱收集+人工修图服务测试，1周内验证    |
| 价值定位错误 | 注重背景融合，字体单一 | 优先增加3-5种字体选择，重点做字体匹配，背景吸取简化  |
| 用户获取不足 | Discord机器人难推广    | 极简网页版+人工修图启动+分享裂变，低成本获取种子用户 |

## 九、最终目标

· MVP可用：用户愿意使用，操作顺畅

· 记录完整开发与技术决策，保证可追溯不返工

· 提供v0.2迭代可靠用户数据与验证结果
